## 二、现有问题 (P0 - 必须修复)

### 2.1 测试覆盖严重不足

**问题描述**:
- `dom-utils`: 测试仅检查函数类型，未验证任何功能
- `regexps`: 完全没有测试文件

**修复方案**:
```typescript
// dom-utils 测试增强示例
describe('getLcShapeElement', () => {
  it('should return null when no element matches', () => {
    document.body.innerHTML = '<div class="other">test</div>'
    expect(getLcShapeElement('non-existent')).toBeNull()

  it('should return element when shape-id matches', () => {
```

### 2.2 错误处理不当

**问题位置**: `crypto-utils/src/index.ts:19-21`

```typescript
// 当前代码 - 吞掉所有错误
} catch {
  return ''
}
```

**问题**: 无法区分解密失败的具体原因，且空字符串可能与正常解密结果冲突

**修复方案**:
```typescript
// 方案 A: 返回 Result 类型
export const decrypt = (word: string, options: AesOptions): Result<string, CryptoError> => {
  try {
    // ... 解密逻辑
    return { ok: true, value: decryptedString }
  } catch (error) {
    return { ok: false, error: CryptoError.InvalidCiphertext }
  }
}

// 方案 B: 抛出具体错误 (推荐)
export class CryptoError extends Error {
  static InvalidKey = class extends CryptoError {
    constructor() { super('Invalid encryption key') }
  }
  static InvalidCiphertext = class extends CryptoError {
    constructor() { super('Invalid ciphertext format') }
  }
}
```

## 三、代码质量优化 (P1 - 建议优化)

### 3.1 Console 调用应替换为标准日志/错误处理

**问题位置**: `tree-utils` 多处使用 `console.error`/`console.warn`

**修复方案**:
```typescript
// 创建可配置的日志器
export interface TreeUtilsOptions {
  logger?: Logger
  onError?: (message: string) => void
}

const defaultLogger = {
  warn: (msg: string) => console.warn('[tree-utils]', msg),
  error: (msg: string) => console.error('[tree-utils]', msg)
}
```

### 3.2 重复代码模式

**问题**: `padZero` 和 `formatTime` 在多处重复实现

**优化方案**:
```typescript
// 提取公共工具函数到单独的包
// @zid-utils/common-utils
export const padZero = (num: number, length: number = 2): string => 
  num.toString().padStart(length, '0')
```

### 3.3 过度复杂的正则表达式

**问题位置**: `regexps` 包中的 IPv6 正则表达式 (L11)

**优化方案**:
```typescript
// 拆分为多行或使用注释
```

## 七、总结

本项目具有良好的 Monorepo 基础架构，但在测试覆盖、错误处理、代码一致性方面存在明显改进空间。建议按照 **P0 → P1 → P2** 的优先级逐步优化，同时预留扩展点以支持未来的功能增长。

核心优化方向:
1. **测试驱动**: 确保每个函数都有充分测试
2. **类型安全**: 完善 TypeScript 类型定义
3. **错误处理**: 使用统一的错误处理机制
4. **命名一致**: 统一函数命名规范
5. **功能模块化**: 提取公共代码，减少重复
